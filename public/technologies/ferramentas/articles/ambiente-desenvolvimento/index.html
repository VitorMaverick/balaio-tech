Como configurei meu ambiente de desenvolvimento

Por Maverick Dev • 14 de janeiro de 2025

Ferramentas

Sabe aquela sensação de abrir o editor e tudo funcionar bonito? Autocompletar certinho, terminal integrado respondendo rápido, atalhos na ponta dos dedos. Não é frescura — um ambiente bem configurado te faz codar mais rápido e com menos frustração. Demorei um tempo ajustando o meu, então vou documentar aqui exatamente o que uso e por quê.

Aviso: esse é o MEU setup. Não é o melhor do mundo, é o que funciona pra mim como estudante que trabalha com web (front e back). Adapta pro teu contexto.

VS Code — o editor

Usei Sublime Text por um tempo, testei o Vim (durou 3 dias), mas o VS Code ganhou disparado. É leve o suficiente, tem extensões pra tudo e a comunidade é gigante. Qualquer dúvida, alguém já passou pelo mesmo problema.

Pra instalar, baixa direto do site oficial ou, no Linux:

# Ubuntu/Debian
sudo apt update
sudo apt install code

# Ou via snap
sudo snap install code --classic

Extensões que não vivo sem

Essa é a parte que faz diferença. O VS Code "pelado" é bom, mas com extensões vira outro nível:

  • ESLint — aponta erros de JavaScript/TypeScript em tempo real. Não espera compilar pra descobrir que esqueceu um ponto e vírgula.
  • Prettier — formata o código automaticamente ao salvar. Nunca mais brigo com indentação.
  • Live Server — abre um servidor local com hot reload. Salva o HTML e o navegador atualiza sozinho.
  • GitLens — mostra quem editou cada linha, quando, e em qual commit. Ótimo em projetos com mais gente.
  • Thunder Client — tipo um Postman dentro do VS Code. Testa APIs sem sair do editor.
  • Material Icon Theme — ícones bonitinhos pra cada tipo de arquivo. Parece besteira mas ajuda a identificar rápido.
  • Error Lens — mostra os erros inline, direto na linha com problema. Não precisa ficar olhando o painel de problemas.
  • Auto Rename Tag — renomeia a tag de fechamento quando você muda a de abertura. Salva tempo com HTML.

Configurações do VS Code

Além das extensões, algumas configurações no settings.json que fazem diferença:

{
    "editor.fontSize": 15,
    "editor.fontFamily": "Fira Code, monospace",
    "editor.fontLigatures": true,
    "editor.formatOnSave": true,
    "editor.defaultFormatter": "esbenp.prettier-vscode",
    "editor.tabSize": 2,
    "editor.wordWrap": "on",
    "editor.minimap.enabled": false,
    "editor.bracketPairColorization.enabled": true,
    "terminal.integrated.fontSize": 14,
    "workbench.colorTheme": "One Dark Pro",
    "files.autoSave": "onFocusChange"
}

O formatOnSave é a configuração mais importante pra mim. Nunca mais perco tempo formatando manualmente. Escrevo o código de qualquer jeito e ao salvar ele fica bonitinho.

Ligatures do Fira Code transformam != em ≠ e => numa setinha. É só estética, mas deixa o código mais legível visualmente.

Terminal — Oh My Zsh

O terminal padrão funciona, mas é sem graça. Instalei o Zsh com Oh My Zsh e a experiência mudou completamente:

# Instalar Zsh
sudo apt install zsh

# Definir como shell padrão
chsh -s $(which zsh)

# Instalar Oh My Zsh
sh -c "$(curl -fsSL https://raw.github.com/ohmyzsh/ohmyzsh/master/tools/install.sh)"

Depois de instalar, configurei no arquivo ~/.zshrc:

# Tema (uso o spaceship, mas powerlevel10k também é ótimo)
ZSH_THEME="spaceship"

# Plugins úteis
plugins=(
    git
    zsh-autosuggestions
    zsh-syntax-highlighting
    node
    npm
)

Os plugins que mais uso:

  • git — atalhos tipo gst (git status), gco (git checkout), gcmsg (git commit -m)
  • zsh-autosuggestions — sugere comandos baseado no histórico. Aperta seta direita pra aceitar.
  • zsh-syntax-highlighting — colore os comandos. Verde se existe, vermelho se tá errado.
Se usa Windows, o PowerShell com Oh My Posh é a alternativa equivalente. Ou instala WSL2 e usa Linux dentro do Windows — é o que muita gente faz.

Node.js — com NVM

Não instala o Node direto do site. Usa o NVM (Node Version Manager). Ele deixa você ter várias versões do Node e alternar entre elas. Isso salva quando um projeto precisa de Node 18 e outro de Node 20:

# Instalar NVM
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.0/install.sh | bash

# Reiniciar o terminal e depois:
nvm install --lts
nvm use --lts

# Ver versão instalada
node --version
npm --version

Depois instalo alguns pacotes globais que uso frequentemente:

npm install -g nodemon    # reinicia servidor automaticamente
npm install -g live-server # servidor estático rápido
npm install -g typescript  # compilador TS global

Git — configuração inicial

Depois de instalar o Git, a primeira coisa é configurar nome e email (são os que aparecem nos commits):

git config --global user.name "Maverick Dev"
git config --global user.email "maverick@email.com"

# Editor padrão (pra mensagens de commit longas)
git config --global core.editor "code --wait"

# Branch padrão como main ao invés de master
git config --global init.defaultBranch main

Também configuro SSH pro GitHub pra não ter que digitar senha toda vez que dou push:

# Gerar chave SSH
ssh-keygen -t ed25519 -C "maverick@email.com"

# Copiar a chave pública
cat ~/.ssh/id_ed25519.pub
# Cola no GitHub → Settings → SSH and GPG keys

Navegador — DevTools é obrigatório

Uso o Chrome como navegador principal de desenvolvimento por causa do DevTools. Algumas coisas que aprendi a usar e não largo mais:

  • Elements: inspecionar e editar HTML/CSS ao vivo. Testo mudanças visuais antes de mexer no código.
  • Console: testar JavaScript rápido, ver erros, debugar com console.log (sim, ainda uso muito).
  • Network: ver todas as requisições HTTP. Fundamental quando tô trabalhando com APIs. Mostra status, tempo, payload.
  • Application: localStorage, sessionStorage, cookies. Tudo que o navegador guarda.
  • Lighthouse: audita performance, acessibilidade e SEO. Roda e vai corrigindo os problemas que aponta.

Extensões do Chrome que ajudam:

  • React Developer Tools — se trabalha com React, é obrigatório.
  • JSON Viewer — formata JSON bonito quando abre uma URL de API no navegador.
  • ColorZilla — pega cores de qualquer elemento da página.

Organização de pastas

Pode parecer bobo, mas ter uma estrutura organizada no computador faz diferença. Minha estrutura:

~/dev/
├── projetos/        # projetos pessoais
├── estudos/         # exercícios e cursos
├── freelas/         # trabalhos freelance
└── sandbox/         # testes rápidos descartáveis

A pasta sandbox é minha favorita. Quando quero testar algo rápido sem criar um projeto formal, jogo lá. Não tem compromisso, posso deletar quando quiser.

Extras que fazem diferença

  • Spotify no fundo: lofi ou trilha sonora de jogo. Parece besteira mas me mantém focado.
  • Segundo monitor: código num, documentação/navegador no outro. Se não tem monitor extra, usa workspaces virtuais do sistema.
  • Pomodoro: 25 min focado, 5 min de pausa. Uso uma extensão do navegador mesmo.
Não gasta uma semana configurando ambiente ao invés de codar. Começa com o básico e vai ajustando conforme sente necessidade. Eu fui adicionando extensão por extensão ao longo de meses, não tudo de uma vez.

Conclusão

Seu ambiente de desenvolvimento é onde você passa horas todo dia. Vale investir um tempo configurando. Mas lembra: a melhor configuração é a que te deixa produtivo, não a mais bonita. Se você codar bem num Notepad++, tá valendo. O importante é que as ferramentas te ajudem, não te atrapalhem.

Meu setup hoje é resultado de meses experimentando. Comecei só com VS Code e terminal padrão, e fui evoluindo. Faz o mesmo: começa simples, identifica o que te irrita ou te atrasa, e busca uma solução. Aos poucos seu ambiente vai ficando com a sua cara.
