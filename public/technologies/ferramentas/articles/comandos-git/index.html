Comandos Git que uso todo dia

Por Maverick Dev • 18 de dezembro de 2024

Ferramentas

Git foi uma daquelas coisas que eu evitei por meses. Ficava fazendo backup manual das pastas, tipo "projeto-v1", "projeto-v2-final", "projeto-v2-final-AGORA-VAI". Até que num trabalho em grupo no curso, não teve jeito: ou aprendia Git ou ia sofrer. Aprendi. E hoje não vivo sem.

Esse artigo é o guia que eu queria ter tido. Sem enrolação, direto ao ponto: os comandos que realmente uso no dia a dia, com explicação de quando e por que usar cada um.

Primeiro: o que é Git?

Git é um sistema de controle de versão. Ele salva "fotos" do seu projeto ao longo do tempo. Se você fez besteira, pode voltar atrás. Se está trabalhando com mais gente, cada um pode mexer em partes diferentes sem conflito. É basicamente um Ctrl+Z profissional pro seu código.

Git é a ferramenta local. GitHub/GitLab são plataformas online que guardam seu repositório na nuvem. Não confunde os dois.

git init — Começando um repositório

Tudo começa aqui. Quando crio um projeto novo, a primeira coisa que faço é:

mkdir meu-projeto
cd meu-projeto
git init

Isso cria uma pasta oculta .git que é onde o Git guarda todo o histórico. A partir daqui, o Git tá observando essa pasta.

Se o projeto já existe no GitHub, ao invés de init eu uso clone:

git clone https://github.com/usuario/repositorio.git

git status — Entendendo o estado atual

Esse é o comando que mais digito no dia. Ele mostra o que mudou, o que tá preparado pro commit e o que o Git tá ignorando:

git status

A saída mostra arquivos em vermelho (modificados mas não staged) e em verde (prontos pro commit). Eu rodo esse comando antes de qualquer outro, tipo um "onde eu tô?".

git add — Preparando as mudanças

O Git tem um conceito de "staging area" — uma área intermediária. Antes de salvar um commit, você escolhe quais mudanças quer incluir:

# Adicionar um arquivo específico
git add index.html

# Adicionar todos os arquivos modificados
git add .

# Adicionar só os arquivos de uma pasta
git add src/

No começo eu usava git add . pra tudo, mas aprendi que às vezes é melhor ser seletivo. Se você mexeu em 5 arquivos mas só quer commitar 2 que são relacionados, adiciona só eles. Commits menores e focados são mais fáceis de entender depois.

git commit — Salvando no histórico

Esse é o "Balaio.tech definitivo". O commit salva uma foto do estado atual dos arquivos que estão na staging area:

# Commit com mensagem inline
git commit -m "adiciona formulário de contato"

# Commit abrindo o editor pra mensagem mais longa
git commit

Dicas pra boas mensagens de commit:

  • Começa com verbo no imperativo: "adiciona", "corrige", "remove"
  • Seja específico: "corrige bug no login" é melhor que "fix"
  • Mantenha curto: até uns 50 caracteres na primeira linha
  • Não comita coisa quebrada: se não compila, não comita
Regra pessoal: se eu não consigo explicar o commit em uma frase curta, provavelmente tô tentando colocar coisa demais num commit só.

git log — Vendo o histórico

Quando preciso ver o que já foi feito:

# Log completo
git log

# Log resumido (uma linha por commit)
git log --oneline

# Log com gráfico de branches (meu favorito)
git log --oneline --graph --all

O --oneline --graph --all é lindo. Mostra as branches como linhas se cruzando, parece um mapa de metrô. Ajuda muito a visualizar o que tá acontecendo no projeto.

git branch — Trabalhando com branches

Branch é tipo uma realidade paralela do seu código. Você cria uma branch, faz suas mudanças lá, e se der certo, junta com a branch principal. Se der errado, joga fora sem afetar nada:

# Ver branches existentes
git branch

# Criar uma nova branch
git branch feature/navbar

# Mudar pra outra branch
git checkout feature/navbar

# Criar e já mudar (atalho que mais uso)
git checkout -b feature/navbar

Minha convenção de nomes:

  • feature/nome — funcionalidade nova
  • fix/nome — correção de bug
  • refactor/nome — reorganização de código

git merge — Juntando branches

Quando a feature tá pronta, hora de juntar com a main:

# Primeiro, volto pra main
git checkout main

# Puxo as atualizações mais recentes
git pull

# Junto a branch da feature
git merge feature/navbar

Se ninguém mexeu nos mesmos arquivos, o merge é automático. Se mexeu... aí vem os temidos conflitos. Não entra em pânico. O Git marca no arquivo exatamente onde o conflito tá:

<<<<<<< HEAD
código que tá na main
=======
código que tá na sua branch
>>>>>>> feature/navbar

Você decide qual versão manter (ou combina as duas), remove as marcações e faz um novo commit. No começo dá medo, mas depois vira rotina.

git push — Mandando pro remoto

Push é o que manda seus commits locais pro GitHub/GitLab:

# Push da branch atual
git push

# Primeiro push de uma branch nova
git push -u origin feature/navbar

# Push da main
git push origin main

O -u (upstream) cria a conexão entre sua branch local e a remota. Depois do primeiro push com -u, os próximos basta git push sozinho.

git pull — Trazendo atualizações

Pull é o inverso do push. Traz as mudanças do remoto pro seu computador:

git pull

Eu sempre dou pull antes de começar a trabalhar. Se o colega fez push ontem à noite, quero ter essas mudanças antes de começar a mexer. Evita conflitos desnecessários.

Meu fluxo de trabalho real

Aqui tá como meu dia a dia funciona com Git:

# 1. Começo do dia: atualizo a main
git checkout main
git pull

# 2. Crio uma branch pra tarefa
git checkout -b feature/card-produto

# 3. Codifico... testo... funciona!

# 4. Verifico o que mudou
git status

# 5. Adiciono e comito
git add .
git commit -m "cria componente de card do produto"

# 6. Mando pro remoto
git push -u origin feature/card-produto

# 7. Abro um Pull Request no GitHub

# 8. Depois de aprovado, faço merge na main

Comandos de emergência

Pra quando as coisas dão errado (e vão dar):

# Desfazer mudanças não commitadas num arquivo
git checkout -- arquivo.txt

# Tirar arquivo da staging area (desfaz o add)
git reset HEAD arquivo.txt

# Voltar pro commit anterior (CUIDADO: perde mudanças)
git reset --hard HEAD~1

# Ver diferenças antes de commitar
git diff
Dica de ouro: o git reset --hard é perigoso. Só use se realmente quer jogar fora o que fez. Na dúvida, usa git stash que guarda as mudanças sem apagar.

.gitignore — O que NÃO versionar

Cria um arquivo .gitignore na raiz do projeto pra dizer pro Git o que ignorar:

node_modules/
.env
dist/
*.log
.DS_Store

Nunca, jamais, suba a pasta node_modules pro GitHub. São milhares de arquivos que qualquer pessoa pode baixar com npm install. E nunca suba arquivos .env com senhas e chaves de API.

Resumindo

Git parece complicado no começo porque tem muitos comandos. Mas no dia a dia, você usa uns 10 comandos e resolve 95% das situações. Minha dica: usa Git desde o primeiro projeto pessoal. Nem precisa GitHub no começo — só o git init e commitar local já te salva. A confiança vem com a prática, e quando você se acostumar, vai se perguntar como vivia sem.
