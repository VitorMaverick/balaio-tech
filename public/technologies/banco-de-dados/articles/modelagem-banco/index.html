Modelagem de banco de dados: o básico que funciona

Por Maverick Dev • 5 de janeiro de 2025

Banco de Dados

Modelagem de banco de dados era o tipo de matéria que eu achava chata até precisar criar um projeto de verdade. Quando tentei fazer um sistema de cadastro sem pensar na estrutura antes, virou um caos. Dados duplicados, tabelas que não faziam sentido, queries impossíveis. Aí entendi: modelar antes de criar é tão importante quanto planejar antes de construir uma casa.

Vou te mostrar os conceitos fundamentais com exemplos práticos. Nada de teoria abstrata demais — só o que você precisa pra começar a estruturar seus bancos de forma decente.

Entidades e atributos

Pensa no seu sistema. Quais são as "coisas" que ele precisa guardar? Cada "coisa" é uma entidade. Num sistema de escola, por exemplo:

  • Aluno — nome, matrícula, email, data de nascimento
  • Professor — nome, especialidade, email
  • Disciplina — nome, carga horária, ementa
  • Turma — código, semestre, horário

Cada entidade vira uma tabela no banco. Os atributos viram as colunas. Simples assim. Vamos criar a tabela de alunos em SQL:

CREATE TABLE alunos (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    matricula VARCHAR(20) UNIQUE NOT NULL,
    email VARCHAR(100) UNIQUE,
    data_nascimento DATE,
    criado_em TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Chave primária (Primary Key)

Toda tabela precisa de uma chave primária. É o campo que identifica cada registro de forma única. Geralmente é um id numérico que incrementa automaticamente:

  • Deve ser única (nunca se repete)
  • Não pode ser nula
  • Não deve mudar ao longo do tempo

Pode usar o id numérico (INT AUTO_INCREMENT) ou UUID. Pra projetos iniciantes, o id numérico resolve bem e é mais fácil de trabalhar.

Relacionamentos — a parte divertida

Entidades raramente existem isoladas. Elas se conectam. E existem três tipos de relacionamento que você vai encontrar em 99% dos casos:

Relacionamento 1:1 (Um para Um)

Cada registro de uma tabela se relaciona com no máximo um registro da outra. Exemplo: cada aluno tem um endereço, e cada endereço pertence a um aluno só.

CREATE TABLE enderecos (
    id INT PRIMARY KEY AUTO_INCREMENT,
    aluno_id INT UNIQUE NOT NULL,
    rua VARCHAR(200),
    cidade VARCHAR(100),
    estado CHAR(2),
    cep VARCHAR(9),
    FOREIGN KEY (aluno_id) REFERENCES alunos(id)
);

O UNIQUE no aluno_id garante que um aluno não tenha dois endereços. Na prática, relacionamentos 1:1 são os menos comuns. Muitas vezes faz sentido colocar tudo na mesma tabela.

Relacionamento 1:N (Um para Muitos)

Esse é o mais comum. Um professor leciona várias disciplinas, mas cada disciplina tem um professor responsável:

CREATE TABLE professores (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    especialidade VARCHAR(100),
    email VARCHAR(100) UNIQUE
);

CREATE TABLE disciplinas (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    carga_horaria INT NOT NULL,
    professor_id INT,
    FOREIGN KEY (professor_id) REFERENCES professores(id)
);

A chave estrangeira (FOREIGN KEY) fica na tabela do lado "muitos". A disciplina aponta pro professor, não o contrário. Esse é um erro comum no começo: colocar a FK no lado errado.

Macete: pergunte "quem pertence a quem?". A disciplina pertence ao professor. Então a disciplina guarda o id do professor.

Relacionamento N:N (Muitos para Muitos)

Aqui complica um pouquinho. Um aluno cursa várias disciplinas, e cada disciplina tem vários alunos. Não dá pra resolver com uma FK simples — precisa de uma tabela intermediária (tabela de junção):

CREATE TABLE matriculas (
    id INT PRIMARY KEY AUTO_INCREMENT,
    aluno_id INT NOT NULL,
    disciplina_id INT NOT NULL,
    semestre VARCHAR(6) NOT NULL,
    nota DECIMAL(4,2),
    FOREIGN KEY (aluno_id) REFERENCES alunos(id),
    FOREIGN KEY (disciplina_id) REFERENCES disciplinas(id),
    UNIQUE(aluno_id, disciplina_id, semestre)
);

A tabela matriculas conecta alunos e disciplinas. Cada linha representa "este aluno está matriculado nesta disciplina neste semestre". O UNIQUE composto impede que o mesmo aluno se matricule duas vezes na mesma disciplina no mesmo semestre.

A tabela de junção também pode ter seus próprios atributos (nota, data de matrícula, frequência). Isso é super útil na prática.

Chave estrangeira (Foreign Key)

A FK é o que cria o vínculo entre tabelas. Ela diz: "o valor nesta coluna deve existir como chave primária na outra tabela". Isso garante integridade referencial — você não consegue matricular um aluno que não existe.

-- Isso vai dar erro se o aluno 999 não existir:
INSERT INTO matriculas (aluno_id, disciplina_id, semestre)
VALUES (999, 1, '2024.1');

E quando deletar um professor? O que acontece com as disciplinas dele? Aí entram as regras de cascata:

-- ON DELETE CASCADE: apaga as disciplinas junto
-- ON DELETE SET NULL: coloca NULL no professor_id
-- ON DELETE RESTRICT: impede de deletar o professor (padrão)

FOREIGN KEY (professor_id) REFERENCES professores(id)
    ON DELETE SET NULL

Normalização: organizando sem exagerar

Normalização é o processo de organizar as tabelas pra evitar redundância. Sem entrar na teoria pesada, as três primeiras formas normais resolvem a maioria dos problemas:

1ª Forma Normal (1FN): cada coluna deve ter um valor atômico (indivisível). Nada de guardar "João, Maria, Pedro" numa coluna só.

-- ERRADO ❌
| id | nome  | telefones          |
|----|-------|-------------------|
| 1  | João  | 99999, 88888      |

-- CERTO ✓ (tabela separada)
CREATE TABLE telefones (
    id INT PRIMARY KEY AUTO_INCREMENT,
    aluno_id INT NOT NULL,
    numero VARCHAR(15) NOT NULL,
    FOREIGN KEY (aluno_id) REFERENCES alunos(id)
);

2ª Forma Normal (2FN): todo atributo não-chave deve depender da chave primária inteira. Em tabelas com chave composta, não pode ter coluna que depende só de parte da chave.

3ª Forma Normal (3FN): não pode ter dependência transitiva. Se A determina B e B determina C, então C não deve estar na mesma tabela que A. Na prática: se o CEP determina a cidade e o estado, separa em outra tabela.

Na real: normaliza até a 3FN e tá bom pra 90% dos projetos. Desnormalização às vezes é necessária por performance, mas no começo foca em organizar direito.

Exemplo completo: mini sistema de escola

-- Esquema completo
CREATE TABLE alunos (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    matricula VARCHAR(20) UNIQUE NOT NULL,
    email VARCHAR(100)
);

CREATE TABLE professores (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE
);

CREATE TABLE disciplinas (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    carga_horaria INT NOT NULL,
    professor_id INT,
    FOREIGN KEY (professor_id) REFERENCES professores(id)
);

CREATE TABLE matriculas (
    id INT PRIMARY KEY AUTO_INCREMENT,
    aluno_id INT NOT NULL,
    disciplina_id INT NOT NULL,
    semestre VARCHAR(6) NOT NULL,
    nota DECIMAL(4,2),
    FOREIGN KEY (aluno_id) REFERENCES alunos(id),
    FOREIGN KEY (disciplina_id) REFERENCES disciplinas(id)
);

-- Consulta: alunos e suas disciplinas
SELECT a.nome AS aluno, d.nome AS disciplina, m.nota
FROM matriculas m
JOIN alunos a ON m.aluno_id = a.id
JOIN disciplinas d ON m.disciplina_id = d.id
WHERE m.semestre = '2024.1'
ORDER BY a.nome;

Dicas finais

  • Desenha antes: pega papel ou usa ferramentas como dbdiagram.io. Visualizar os relacionamentos antes de escrever SQL economiza muito retrabalho.
  • Nomes no plural ou singular? Eu uso plural pras tabelas (alunos, disciplinas) e singular pro que é referência (aluno_id). Mas o importante é ser consistente.
  • Sempre tenha um id: mesmo que pareça desnecessário, ter um id numérico como PK facilita joins e referências.
  • Pense nas queries: quando estiver modelando, imagina quais perguntas o sistema vai fazer. "Quais alunos estão na disciplina X?" Se a estrutura não permite essa query facilmente, repensa o modelo.

Modelagem é uma habilidade que melhora com prática. No meu primeiro projeto errei feio, no segundo errei menos, e agora já consigo pensar numa estrutura razoável antes de abrir o MySQL. Se você tá começando, pega um sistema simples (biblioteca, loja, escola) e tenta modelar. Vai errar, vai refazer — e é assim que aprende.
