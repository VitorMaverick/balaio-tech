Minha primeira API REST: o que aprendi

Por Maverick Dev • 10 de dezembro de 2024

Back-end

Cara, criar minha primeira API REST foi um divisor de águas. Sério. Antes disso eu ficava preso no front-end, fazendo tudo no navegador, e não entendia como os dados chegavam lá. Quando finalmente sentei pra estudar o back-end, percebi que não era esse bicho de sete cabeças que eu imaginava.

Vou compartilhar aqui tudo o que aprendi nesse processo, desde o conceito até o código funcionando. Se você tá começando agora, fica comigo que vai fazer sentido.

Mas afinal, o que é uma API REST?

Pensa assim: uma API (Application Programming Interface) é um garçom. O front-end é o cliente que faz o pedido, o back-end é a cozinha que prepara. A API é quem leva o pedido e traz a resposta. Simples assim.

Agora, REST (Representational State Transfer) é um conjunto de regras que esse garçom segue. São convenções que todo mundo combinou de usar pra que as APIs funcionem de forma previsível. As principais ideias são:

  • Recursos: tudo é um recurso (usuários, produtos, posts). Cada recurso tem uma URL própria.
  • Métodos HTTP: usamos verbos como GET, POST, PUT e DELETE pra indicar o que queremos fazer.
  • Stateless: cada requisição é independente. O servidor não lembra quem você é entre uma requisição e outra.
  • JSON: o formato padrão pra enviar e receber dados. É leve e fácil de ler.

Os métodos HTTP na prática

Essa parte me confundiu no começo, mas depois ficou natural. Pensa numa lista de tarefas:

  • GET /tarefas → busca todas as tarefas
  • GET /tarefas/1 → busca a tarefa com id 1
  • POST /tarefas → cria uma nova tarefa
  • PUT /tarefas/1 → atualiza a tarefa com id 1
  • DELETE /tarefas/1 → deleta a tarefa com id 1

Percebe o padrão? A URL é sempre a mesma (/tarefas), o que muda é o método HTTP. Isso é REST sendo bonito.

Montando o projeto com Node.js e Express

Escolhi Node.js porque já sabia JavaScript do front-end, então a curva de aprendizado foi menor. E o Express é um framework minimalista que facilita muito a criação de rotas. Vamos ao setup:

# Criar a pasta do projeto e inicializar
mkdir minha-api
cd minha-api
npm init -y

# Instalar o Express
npm install express

Depois de instalar, criei um arquivo server.js na raiz do projeto. Esse é o coração da nossa API:

const express = require('express');
const app = express();
const PORT = 3000;

// Middleware pra entender JSON no body das requisições
app.use(express.json());

// Banco de dados fake (array na memória)
let tarefas = [
    { id: 1, titulo: 'Estudar Node.js', concluida: false },
    { id: 2, titulo: 'Fazer projeto do curso', concluida: true }
];

// GET - Listar todas as tarefas
app.get('/tarefas', (req, res) => {
    res.json(tarefas);
});

// GET - Buscar tarefa por ID
app.get('/tarefas/:id', (req, res) => {
    const tarefa = tarefas.find(t => t.id === Number(req.params.id));
    if (!tarefa) {
        return res.status(404).json({ erro: 'Tarefa não encontrada' });
    }
    res.json(tarefa);
});

// POST - Criar nova tarefa
app.post('/tarefas', (req, res) => {
    const novaTarefa = {
        id: tarefas.length + 1,
        titulo: req.body.titulo,
        concluida: false
    };
    tarefas.push(novaTarefa);
    res.status(201).json(novaTarefa);
});

// PUT - Atualizar tarefa
app.put('/tarefas/:id', (req, res) => {
    const tarefa = tarefas.find(t => t.id === Number(req.params.id));
    if (!tarefa) {
        return res.status(404).json({ erro: 'Tarefa não encontrada' });
    }
    tarefa.titulo = req.body.titulo || tarefa.titulo;
    tarefa.concluida = req.body.concluida ?? tarefa.concluida;
    res.json(tarefa);
});

// DELETE - Remover tarefa
app.delete('/tarefas/:id', (req, res) => {
    const index = tarefas.findIndex(t => t.id === Number(req.params.id));
    if (index === -1) {
        return res.status(404).json({ erro: 'Tarefa não encontrada' });
    }
    tarefas.splice(index, 1);
    res.status(204).send();
});

app.listen(PORT, () => {
    console.log(`Servidor rodando em http://localhost:${PORT}`);
});

Pra rodar, é só digitar node server.js no terminal. Pronto, sua API tá no ar!

Testando a API

No começo eu não sabia como testar sem ter um front-end. Aí descobri que dá pra usar o próprio terminal com curl, ou ferramentas como Postman e Insomnia. Eu gosto do Insomnia porque é mais visual. Mas aqui vão exemplos com curl:

# Listar todas as tarefas
curl http://localhost:3000/tarefas

# Criar uma nova tarefa
curl -X POST http://localhost:3000/tarefas \
  -H "Content-Type: application/json" \
  -d '{"titulo": "Escrever artigo pro blog"}'

# Atualizar uma tarefa
curl -X PUT http://localhost:3000/tarefas/1 \
  -H "Content-Type: application/json" \
  -d '{"concluida": true}'

# Deletar uma tarefa
curl -X DELETE http://localhost:3000/tarefas/2

Status codes: os números importam

Uma coisa que eu ignorava no começo eram os códigos de status HTTP. Mas eles são fundamentais pra comunicar o que aconteceu:

  • 200 — OK, deu tudo certo
  • 201 — Created, recurso criado com sucesso
  • 204 — No Content, deu certo mas não tem nada pra retornar (tipo no DELETE)
  • 400 — Bad Request, o cliente mandou algo errado
  • 404 — Not Found, recurso não existe
  • 500 — Internal Server Error, algo quebrou no servidor

Retornar o status certo faz diferença porque o front-end usa esses códigos pra decidir o que mostrar pro usuário.

O que é esse tal de JSON?

JSON (JavaScript Object Notation) é basicamente a língua que front e back usam pra se entender. É um formato de texto que representa dados de forma estruturada:

{
    "id": 1,
    "titulo": "Estudar Node.js",
    "concluida": false,
    "tags": ["estudo", "backend"]
}

É parecido com um objeto JavaScript, mas com algumas diferenças: as chaves precisam estar entre aspas duplas e não aceita funções. Praticamente toda API moderna usa JSON, então vale dominar.

Erros que cometi (pra você não repetir)

Vou listar os erros que me fizeram perder tempo:

  • Esquecer o express.json(): sem esse middleware, o req.body vem undefined. Passei meia hora debugando isso.
  • Não converter o params pra número: req.params.id vem como string. Se comparar com === num número, nunca vai achar.
  • Esquecer o return antes do res.status(404): sem o return, o código continua executando e pode dar erro de "headers already sent".
  • Usar a mesma porta que outro processo: se der EADDRINUSE, é porque já tem algo rodando naquela porta. Mata o processo ou muda a porta.

Próximos passos

Essa API é só o começo. Ela guarda dados na memória, então quando o servidor reinicia, perde tudo. Os próximos passos naturais são:

  • Conectar num banco de dados de verdade (MongoDB, PostgreSQL)
  • Adicionar validação dos dados de entrada
  • Implementar autenticação (JWT é o mais comum)
  • Organizar o código em pastas (routes, controllers, models)
  • Adicionar tratamento de erros global
Dica: não tenta fazer tudo de uma vez. Faz funcionar primeiro, depois melhora. Meu primeiro código era feio, mas rodava. E isso é o que importa no começo.

Criar essa API me deu confiança pra entender como aplicações web funcionam de verdade. Quando voltei pro front-end e fiz um fetch() pra minha própria API, a sensação foi de "agora sim eu entendo o ciclo completo". Recomendo demais começar por um CRUD simples assim. Vai por mim, vale cada minuto.
